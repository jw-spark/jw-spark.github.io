1. 数据安全与隐私风险

1.1 训练数据安全风险

训练数据是构建和塑造AI模型能力与行为的基石， 其安全性直接关系到AI系统的可信度和可靠性。此类风险聚焦于训练数据本身可能遭受的恶意篡改（数据污染/投毒） 、其中包含的敏感个人信息或商业机密的意外泄露、数据来源的不合规以及数据中固有的偏见被模型学习并放大。未能有效管理训练数据安全风险，将从源头上引入重大缺陷，可能导致AI模型产生错误的决策、 侵犯用户隐私、 违反法律法规或输出具有歧视性的结果， 其影响深远且难以彻底纠正。

1.1.1 数据污染/投毒 (Data Poisoning/Contamination)
概要说明
攻击者通过向训练数据集中注入精心构造的恶意样本，使得AI模型学习到错误的模式或隐藏的后门，从而在后续的推理阶段针对特定输入产生错误的输出结果或执行⾮预期的恶意操作。
相关者角色 (RACI)
R 数据工程团队、AI模型研发团队
A AI治理委员会 / ⾸席数据官 (CDO)
C 安全团队、法务合规团队、业务部门负责人
I AI产品负责人、IT运维团队负责人
控制措施
  • n 管理制度：制定严格的数据来源审查和验证流程， 确保数据源的纯净性和可信度；建立持续的数据质量监控机制， 对训练数据进行有效性评估和异常检测； 实施完整的数据⾎缘追踪，确保数据处理过程的透明度和可追溯性。 n 架构设计: 设计健壮的数据预处理管道， 集成数据清洗、 去噪、 异常值检测等功能模块；考虑采用联邦学习、 差分隐私等技术， 在数据收集和训练阶段减少对原始敏感数据的直 接依赖，降低投毒风险。 n 安全功能: 部署数据投毒检测算法， 例如基于统计分析、 模型行为分析或离群点检测的方法；对来自外部提供商或众包平台的数据进行隔离环境下的验证和严格的抽样检查。
验证措施
  • n 技术验证：通过模拟不同类型的投毒攻击（如标签翻转、后门样本注入）来测试数据清洗和检测机制的有效性； 利用数据分析工具对训练数据集进行深入的统计分析， 以检测异常的数据分布或可疑的数据模式； 对模型进行全面的鲁棒性测试， 评估其在面对轻微数据扰动或潜在污染时的行为稳定性。 n 流程审计: 定期审计数据采集、 处理和标注流程， 确保所有操作均符合既定的安全规范和质量标准；对数据⾎缘记录进行细致审查，验证数据处理过程的合规性和准确性。

1.1.2 训练数据隐私泄露 (Training Data Privacy Leakage)
概要说明
AI模型在训练过程中接触到的包含个人身份信息（PII）、 商 业 秘 密 或 其 他 敏 感 数 据 ， 这 些 数据在模型训练完成后， 可能通过模型的推理输出、 成员推断攻击、 模型逆向工程等技术手段被⾮授权方获取，从而导致隐私泄露。
相关者角色 (RACI)
R 数据工程团队、AI模型研发团队、隐私工程团队、法务合规团队
A ⾸席隐私官 (CPO) / 数据保护官 (DPO)
C AI安全团队、业务部门负责人
I AI产品负责人、IT运维团队负责人
控制措施
  • n 管理制度：制定并严格执行数据分类分级制度，明确敏感数据的识别标准和处理要求； 全面实施数据最小化原则，确保仅收集和使用与AI模型训练目标直接相关的最少必要数据；建立完善的数据脱敏流程，对训练数据中的PII和其他敏感信息采用匿名化、假名化或差分隐私等技术进行处理。 n 架构设计: 设计并采用⽀持先进隐私保护技术 （如联邦学习、 安全多方计算、 同态加密）的AI模型训练架构；在数据存储、处理和传输的各个环节采用端到端的强加密措施。 n 安全功能: 集成自动化的数据发现和分类工具，⾼效识别训练数据中的各类敏感信息；部署数据泄露防护 （DLP）系 统 ，实 时 监 控 并 阻 ⽌ 训 练 数 据 在 不 安 全 环 境 中 的 异 常 流 转 ；开发或引入专业的模型隐私评估工具，检测模型是否存在记忆特定训练样本或泄露训练数据分布特征的风险。
验证措施
  • n 技术验证：主动进行成员推断攻击、 属性推断攻击等隐私攻击的模拟测试， 量化评估模型的隐私泄露风险等级； 使用差分隐私验证工具， 检验所采用的隐私保护技术的实际有效性；对经过脱敏处理的数据进行分析， 确保敏感信息已被充分去除， 同时评估数据可用性是否受到可接受范围内的影响。 n 合规审计: 定期开展隐私影响评估（PIA）； 审 计 数 据 处 理 活 动 是 否 全 面 符 合 适 用 的 隐 私法规（如《个人信息保护法》、《数据安全法》、GDPR、CCPA）的要求。

1.1.3 数据来源与合规性风险
概要说明
用于训练AI模型的数据其来源渠道不合法、未经数据所有者或相关权利方的适当授权，或者不符合适用的数据保护及知识产权等法律法规的要求， 从而可能引发法律诉讼、 监管处罚和企业声誉损害。
相关者角色 (RACI)
R 数据采购与供应商管理团队、法务合规团队、数据治理团队
A ⾸席合规官 (CCO) / ⾸席数据官 (CDO)
C AI模型研发团队、业务部门负责人
I AI产品负责人、内部审计部门
控制措施
  • n 管理制度：建立严格的数据来源审查和尽职调查流程， 确保所有数据均通过合法合规渠道获取；制定详细的数据使用合规清单， 并对照相关法律法规进行核查；对数据提供商进行严格的资质审核，并在合同中明确数据使用权和合规责任。 n 架构设计: 设计能够清晰追溯数据来源的数据管道和元数据管理系统； 建立数据合规标签体系，对不同来源和合规等级的数据进行分类管理。 n 安全功能: 集成自动化工具， 辅助检查数据集中是否包含特定地理区域的受限数据或受版权保护的内容。
验证措施
  • n 技术验证：对训练数据集进行抽样检查， 核实其来源的合法性和相关许可信息；使用自动化工具扫描数据，识别潜在的版权冲突或法规遵循问题。 n 流程审计: 定期审计数据采购、 接收和使用流程， 确保全面符合内部制定的合规政策和外部的法律法规要求； 审查与数据提供商签订的合同条款， 确保数据使用权限和责任明确。
参考文献
《数据安全法》、《个人信息保护法》; GDPR; CCPA; (相关行业的特定数据法规)
1.1.4 数据偏见与歧视 (Data Bias & Discrimination)
概要说明
训练AI模型所使用的数据中，由于历史原因、采样偏差或标注问题，潜藏着针对特定人群（如基于种族、性别、年龄、地域等）的系统性偏见，导致训练出的AI模型在决策或输出时，会延续甚至放大这些偏见，产生不公平或歧视性的结果。
相关者角色 (RACI)
R AI模型研发团队、数据工程团队、AI伦理委员会/专家、用户体验设计师
A AI伦理负责人 / AI治理委员会
C 法务合规团队、业务部门负责人、人力资源部门（若相关）
I AI产品负责人、企业社会责任部门
控制措施
  • n 管理制度：制定并推行企业AI伦理准则和反歧视政策，明确公平性和无偏见原则；在数据收集和标注阶段，特别关注数据来源的多样性和不同群体的代表性，避免采样偏差；建立完善的数据偏见评估和缓解流程，贯穿数据处理和模型开发全生命周期。 n 架构设计: 优先选择或设计具有较⾼可解释性的AI模型，以便于分析和定位偏见的来源；在数据预处理阶段，研究并采用先进的去偏算法（如重采样、重加权、对抗性去偏等） 。 n 安全功能: 开发或集成专业的偏见检测工具， 对训练数据、 模型中间表示和最终输出进行全面的偏见评估；在模型训练过程中引入公平性约束作为优化目标之一。
验证措施
  • n 技术验证：使用标准化的公平性度量指标（如统计均等差异、机会均等差异等）对训练数据和模型进行量化评估； 进行细致的亚组分析， 比较模型在不同受保护群体上的性能表现是否存在显著差异；邀请具有不同文化背景和视角的测试人员对模型输出进行主观评估，识别潜在的隐性偏见。 n 流程审计: 审查数据收集、 标注指南和流程， 确保已充分考虑并采取措施减轻偏见风险；定期由企业内部的AI伦理委员会或独立的第三方机构对AI系统进行偏见审计和评估。
参考文献
NIST AI RMF (Bias in AI)

1.2 输入/输出数据安全风险（交互过程中的数据风险）

1.2.1 提示词注入/恶意利用 (Prompt Injection/Malicious Prompt)
概要说明
攻击者通过在用户输入（即提示词）中嵌入精心构造的指令、代码或特殊字符序列，欺骗或操纵大型语⾔模型绕过其预设的安全护栏、 偏离原始任务目标， 从而执行⾮预期的操作， 例如泄露系统内部的敏感信息、执行恶意代码或生成有害、误导性的内容。
相关者角色 (RACI)
R AI应用研发团队、AI模型研发团队（负责安全护栏） 、安全测试团队
A AI安全负责人 / 应用负责人
C 安全架构师
I 运维团队、AI产品负责人
控制措施
  • n 管理制度：将提示词注入攻击作为AI应用安全测试重点；用户输入安全意识培训。 n 架构设计: 设计多层防御机制（输入净化、输出编码、上下文隔离） ；参数化查询或结构化输入。 n 安全功能: 部署输入验证和净化引擎；使用提示词注入检测模型/规则；模型输出后处理移除恶意内容；限制模型执行⾼风险操作。
验证措施
  • n 技术验证：针对性提示词注入攻击测试；自动化提示词注入测试工具；红队演练。 n 代码审计: 审查输入处理和模型调用代码。

相关者角色 (RACI)

A 物联⽹(IoT)/边缘计算安全负责人
C AI模型研发团队（负责模型鲁棒性）
I AI产品负责人、业务运营团队
控制措施

5. AI运行环境与基础设施安全风险

5.1 物理环境安全风险

AI系统的运行离不开坚实的物理环境⽀撑，然而无论是传统的数据中心、新兴的边缘计算节点， 还是⼴泛分布的端侧设备， 都面临着不同程度的物理安全威胁。 此类风险源于未经授权的物理访问、 设备盗窃或损坏、 自然灾害或环境因素的破坏、 以及针对特定部署场景（如边缘和端侧）的物理篡改或近距离攻击。物理环境的失陷可能直接导致AI系统硬件损 坏、数据丢失、敏感信息泄露或服务中断，是AI安全防护中不可或缺的基础层面。

5.1.1 数据中心/服务器物理安全
概要说明
托管AI模型、 数据和⽀撑应用的传统数据中心或本地服务器机房， 由于物理访问控制不⾜、环境监控缺失、 自然灾害防护不力或内部人员恶意破坏等原因， 导致服务器硬件被盗窃、 损坏、遭受⾮授权物理访问或关键基础设施（如电力供应、空气调节与制冷系统）发生中断，从而严重影响AI系统的持续可用性、数据完整性和信息保密性。
相关者角色 (RACI)
R 数据中心运营团队、物理安全团队、设施管理部门
A ⾸席信息官 (CIO) / IT基础架构负责人/CISO
C IT运维团队、灾难恢复与业务连续性团队、供应商（提供数据中心服务）
I 各业务部门负责人、审计团队、管理层
控制措施
  • n 管理制度：制定并严格执行数据中心的物理访问控制策略和多级授权流程； 定期组织物理安全风险评估、 应急响应演练和安全检查；建立并维护完善的环境监控系统 （如温湿度、烟感、漏水、消防等）和相应的应急处置预案。 n 架构设计: 在数据中心选址时充分考虑地理位置的安全性 （如远离自然灾害⾼发区、 交通便利但相对隐蔽） ；设计并实施多重物理防线，包括周界防护、区域隔离、机柜锁定等。 n 安全功能: 全面部署视频监控系统（CCTV）、 入 侵 检 测 与 报 警 系 统 、 生 物 识 别 或 多 因 素认证的门禁控制系统； 对服务器机柜进行严格的上锁管理和出入库登记； 实施严格的访客登记、身份验证和全程陪同制度。
验证措施
  • n 技术验证与演练：定期进行物理渗透测试， 模拟攻击者尝试通过各种手段未授权进入数 据中心或接触关键服务器； 全面测试报警系统、 监控系统和门禁控制系统的实际有效性和联动响应能力。 n 流程审计与检查: 定期审计物理访问日志、 安保人员的巡检记录和设备维护记录；审查应急预案的完整性、可操作性和与实际情况的符合度。

5.1.2 边缘节点物理安全
概要说明

验证措施
  • n 技术验证与模拟攻击：模拟针对不同部署环境下的边缘计算节点的各种典型物理破坏、⾮法接入 （如通过调试接⼝） 或恶意篡改尝试， 全面测试其所部署的各项物理防护措施的实际有效性； 严格测试设备自身的篡改检测功能和远程告警机制的响应速度、 准确性和可靠性。 n 现场巡检与环境评估: 定期或根据风险评估结果不定期对部署在各个地理位置的边缘计算节点进行全面的物理安全状况巡检和周边环境风险评估，确保防护措施持续有效。

概要说明

5.2 网络环境安全风险

⽹络是AI系统信息交互和数据流动的命脉， 其安全性直接关系到AI系统的整体稳定和数据保密。 此类风险涵盖了从不必要的⽹络服务暴露和攻击面扩大、 ⽹络隔离与访问控制不⾜、使用不安全的⽹络协议或配置错误，到针对性的分布式拒绝服务（DDoS）攻击、端-边-云复杂通信链路中潜在的劫持与窃听， 以及API安全⽹关和Web应用防火墙（WAF）防 护策略的缺失或不⾜。⽹络层面的任何脆弱性都可能为攻击者提供入侵途径，威胁AI系统的正常运行和数据安全。

5.2.1 开放的暴露面和攻击面
概要说明
AI系统及其⽀撑组件在⽹络上开放了不必要的服务端⼝、管理接⼝或API端点，或者这些暴露面缺乏⾜够的身份认证、 授权和安全加固措施， 从而形成了薄弱的攻击面， 容易被攻击者扫描发现并利用来进行未授权访问、数据窃取、拒绝服务攻击或进一步渗透。
相关者角色 (RACI)
R ⽹络安全团队、系统管理员、DevOps团队、云安全工程师
A CISO / IT架构负责人/AI平台负责人
C 应用研发团队、安全运营中心(SOC)、渗透测试团队
I 审计团队、合规团队
控制措施
  • n 管理制度：制定并严格执行端⼝和服务开放的最小化策略 （默认拒绝所有， 按需允许） ；建立定期的⽹络暴露面扫描、 评估和收敛流程； 对所有面向公⽹或重要内部⽹络的服务进行严格的安全审查和风险评估。 n 架构设计: 在⽹络架构层面， 默认关闭所有不必要的服务端⼝和管理接⼝；将必须面向公⽹的服务和API端点合理地部署在经过安全加固的DMZ（Demilitarized Zone，隔离区）区域，并与内部核心⽹络进行严格隔离。 n 安全功能: 全面部署下一代防火墙（NGFW）或Web应用防火墙（WAF）， 对 所 有 进 出⽹络的流量进行精细化的访问控制和深度包检测；实施⽹络入侵检测/防御系统（NIDS/NIPS）来实时监控和阻⽌可疑的⽹络活动；对所有暴露的服务和API接⼝强制要求强身份认证（如MFA、客户端证书认证）和基于最小权限原则的授权。
验证措施
  • n 技术验证：定期进行全面的外部和内部⽹络扫描（如使用Nmap、Shodan等工具） ，主动识别所有开放的端⼝、 运行的服务及其版本信息；组织专业的渗透测试团队， 模拟攻击者尝试利用已发现的暴露面进行攻击和渗透。 n 配置审计: 定期审查所有防火墙、路由器、交换机以及云平台⽹络安全组（Security Group）等⽹络设备的配置，确保其严格符合最小开放原则和既定的安全策略。

概要说明

C AI平台团队、应用研发团队、数据治理团队
I 审计团队、合规团队

  • n 管理制度：制定明确的⽹络安全域划分标准和跨域访问控制策略，确保AI相关的各类⽹络（如开发⽹、测试⽹、生产⽹、管理⽹、数据存储⽹等）与其他企业业务⽹络之间实现有效的逻辑和物理隔离。 n 架构设计: 全面采用VLAN（虚拟局域⽹） 技术、 防火墙、 路由器ACL（访问控制列表） 、以及更先进的微隔离（Microsegmentation）或软件定义⽹络（SDN）技术来实现⽹络分段和精细化隔离；在⽹络访问控制层面严格实施基于零信任（Zero Trust）原则的架构和策略，即默认不信任任何⽹络流量，所有访问均需经过严格认证和授权。 n 安全功能: 在不同安全域的边界部署⾼性能的防火墙和入侵防御系统（IPS）； 对 所 有 关键⽹络节点和AI服务接⼝实施严格的、基于最小权限原则的访问控制列表（ACL）和安全组策略。

5.2.3 不安全的⽹络协议与配置
概要说明

I 审计团队、应用所有者
控制措施

验证措施
  • n 技术验证：使用专业的协议分析工具（如Wireshark配合特定插件）对⽹络流量进行深度捕获和分析， 检测是否存在不安全协议的使用情况；利用⽹络设备配置扫描工具 （如Nessus、OpenVAS）对所有⽹络设备进行全面的安全配置漏洞扫描。 n 配置审计: 定期对照行业安全基线（如CIS Benchmarks）和企业内部安全策略，对所有⽹络设备的配置参数进行严格审查和验证。

5.2.5 端-边-云通信链路安全
概要说明
在涉及端侧设备、边缘计算节点和云平台的多层AI架构中，各层级之间的数据传输和控制信令通信链路， 由于可能跨越公共⽹络、 使用多样化的无线或有线技术、 以及涉及不同安全域的边界，面临被攻击者窃听、篡改、劫持（如中间人攻击） 、重放或拒绝服务的风险，从而可能导致敏感数据泄露、模型被污染、控制指令被伪造或整个AI系统功能失效。
相关者角色 (RACI)
R ⽹络安全团队、端侧/边缘/云平台研发团队、安全架构师
A CISO / ⽹络架构负责人/AI平台负责人
C 应用研发团队（定义通信需求） 、安全运营中心(SOC)、硬件供应商（涉及通信模块）
I 产品负责人、运维团队
控制措施
  • n 管理制度：强制要求所有端-边-云之间的敏感数据通信和控制信令传输必须使用当前推荐的、经过验证的强加密协议（如TLS 1.3、DTLS 1.2/1.3、IPsec）； 建 立 并 严 格 执 行通信密钥和数字证书的全生命周期管理流程， 包括安全的密钥生成、 分发、 存储、 定期轮换和及时吊销。 n 架构设计: 在可行的情况下，为敏感的边缘节点或特定的端-云数据流部署VPN（虚拟专用⽹络） 、SD-WAN（软件定义⼴域⽹）或专用的物理/逻辑⽹络连接，以增强通信链路的隔离性和保密性；在通信的各个端点（端设备、边缘节点、云服务）实施严格的双向身份认证机制 （例如， 基于X.509客户端证书的mTLS）， 确 保 通 信 双 方 的 身 份 都 是 可信的。 n 安全功能: 在关键的边缘⽹关、 云端入⼝以及重要的端侧设备上部署轻量级或全功能的入侵检测/防御系统 （IDS/IPS）， 实 时 监 控 异 常 通 信 流 量 和 潜 在 的 ⽹ 络 攻 击 行 为 ； 当 使 用消息队列（MQ）、MQTT、CoAP等物联⽹或分布式系统常用的中间件进行通信时，必须确保其自身的通信安全配置（如TLS加密传输、基于用户名/密码或令牌的访问凭证管理、细粒度的主题权限控制等）得到正确且安全的实施。
验证措施
  • n 技术验证与渗透测试：进行有针对性的中间人攻击（MitM）模拟，尝试拦截、解密、篡改或重放端-边-云之间的通信数据， 全面检验加密保护、 身份认证和完整性校验机制的实际有效性；使用专业的⽹络嗅探和协议分析工具（如Wireshark）对实际通信流量进行深度捕获和分析， 检查其加密强度、 协议合规性以及是否存在敏感信息明文传输的情况； 对所有通信端点的数字证书有效性、 信任链完整性以及所使用的加密套件和协议版本进行严格的检查和评估。 n 配置审计与审查: 定期对VPN配置、防火墙规则、边缘⽹关安全策略、消息中间件安全配置以及云平台的⽹络安全组（Security Group）或⽹络ACL（NACL）等进行严格的审查和审计，确保其全面符合既定的安全标准和行业最佳实践。

5.2.6 API安全⽹关与WAF防护不⾜
概要说明
用于保护AI服务API接⼝的API安全⽹关或Web应用防火墙（WAF）， 由 于 其 自 身 配 置 不 当、规则库更新不及时、缺乏针对AI特有攻击模式（如大规模提示词注入、模型资源耗尽型查询） 的防护策略， 或者其检测和拦截能力不⾜， 导致无法有效抵御针对AI API的各类已知和新兴威胁，使得AI应用面临被攻击的风险。
相关者角色 (RACI)
R API安全团队、应用安全团队、DevOps/SRE团队、安全运营中心(SOC)
A 应用负责人/CISO/⾸席架构师
C 安全架构师、AI应用研发团队（提供API规范和预期流量模式）
I AI应用研发团队、运维团队
控制措施
  • n 管理制度：制定并定期更新API安全策略和WAF防护规则库，确保能够覆盖最新的OWASP API Security Top 10风险以及针对AI和大型语⾔模型的特有攻击向量；建立API⽹关和WAF的配置基线和变更管理流程。 n 架构设计: 确保所有面向外部或内部⾼风险区域的AI相关API都必须通过API安全⽹关和WAF进行统一的接入和安全防护；API⽹关应与IAM系统紧密集成，实现对API的强身份认证和细粒度授权。 n 安全功能: 在API⽹关和WAF层面精细化配置API的认证授权策略、请求参数校验规则、请求/响应内容过滤、以及基于IP、用户、API密钥等多维度的速率限制和并发控制；针对OWASP API Security Top 10中列出的风险（如失效的对象级别授权、批量分配、安全配置错误等）和针对AI模型的特有攻击（如提示词注入、模型资源耗尽）定制专门的防护规则和检测逻辑；启用API流量的深度分析和异常行为检测功能。
验证措施
  • n 技术验证与渗透测试：定期针对受API⽹关和WAF保护的AI API接⼝进行全面的渗透测试和有针对性的绕过尝试， 评估其防护策略的有效性和健壮性； 使用专业的API安全测试工具（如Postman配合安全测试脚本、专门的API模糊测试工具）对API进行自动化安全扫描。 n 规则有效性评估与优化: 定期回顾和评估现有API安全策略和WAF规则的覆盖率、准确性（误报率和漏报率） ，并根据最新的威胁情报、攻击趋势和业务变化进行持续优化和调整。

5.3 计算环境安全风险（操作系统、虚拟化、容器）

(操作系统、虚拟化、容器) AI模型的训练、 推理和管理任务均在特定的计算环境中执行， 这些环境（包括底层操作系统、虚拟化平台、容器技术）自身的安全性至关重要。此类风险主要源于操作系统存在的已知或未知漏洞、 不安全配置、 虚拟化或容器环境的逃逸与隔离突破、 使用包含漏洞的容器镜像、容器编排工具配置不当，以及在端侧和边缘环境中特有的操作系统和固件安全问题，乃至可信执行环境（TEE）的缺失或错误配置。计算环境的任何安全缺陷都可能被攻击者利用，从而获取系统控制权，危及AI模型的完整性和数据的机密性。

控制措施
  • n 管理制度：建立操作系统安全基线和加固标准；实施严格的补丁管理流程；最小化安装原则。 n 安全功能: 自动化补丁管理工具；配置管理工具（如Ansible, Chef, Puppet）； 主 机 入 侵检测/防御系统（HIDS/HIPS）。

5.4 运行时依赖与库安全风险

现代AI系统⾼度依赖于各种开源AI框架（如TensorFlow, PyTorch）以及大量的第三方库和系统组件来⽀持其复杂的功能。此类风险源于这些运行时依赖项自身可能存在的安全漏洞，例如AI框架中未被及时修复的缺陷、⼴泛使用的第三方库中包含的已知漏洞，或因不安全的序列化/反序列化操作而引入的风险。攻击者一旦成功利用这些依赖项的脆弱性，就可能执行任意代码、 窃取敏感数据、 导致服务拒绝， 或对模型的训练和推理过程造成严重⼲扰，凸显了软件供应链安全在AI运行时的重要性。

5.4.1 AI框架漏洞
概要说明
AI模型所依赖的深度学习框架 （如TensorFlow, PyTorch, Keras等） 自身可能存在安全漏洞，例如缓冲区溢出、 整数溢出、 资源管理不当或不安全的反序列化等。 攻击者可能利用这些框架漏洞来执行任意代码、导致拒绝服务、窃取模型或数据，或影响模型的训练和推理过程。

5.4.3 序列化/反序列化漏洞
概要说明

5.5 资源管理与隔离风险（多租户环境）

在多租户共享的AI平台或边缘计算环境中，对计算、存储、⽹络等核心资源的有效管理和严格隔离是保障各租户安全、公平使用和成本控制的关键。此类风险主要体现在计算/存储等资源未能实现有效的逻辑或物理隔离，导致一个租户的活动可能⾮预期地影响或访问其他租户的数据或应用；以及未能为每个租户或应用设置合理且得到严格执行的资源使用配额和操作限制，使得某些行为不受约束的实体过度消耗共享资源，引发“吵闹的邻居”效应，甚至导致整个平台的拒绝服务或产生不可控的运营成本。

5.5.1 计算/存储资源隔离不当
概要说明
在多租户的AI平台或共享的边缘计算环境中， 不同租户或AI应用共享底层的物理或虚拟化计算、 存储、 ⽹络等基础设施资源。 如果这些共享资源之间的逻辑或物理隔离机制存在设计缺陷、实现漏洞或配置错误，一个租户的恶意行为（如资源耗尽攻击、横向渗透尝试）或意外错误（如程序Bug导致资源泄漏）可能会影响到其他租户的AI应用性能、数据安全或整体服务可用性，破坏了多租户环境应有的隔离性和公平性。

C 安全团队、合规审计团队、租户代表/客户⽀持
I 审计团队、所有平台租户
控制措施
  • n 管理制度：制定并严格执行明确的多租户资源隔离策略和技术标准， 确保不同租户之间的资源在逻辑上和（如果可能）物理上都得到有效隔离。 n 架构设计: 优先采用具有强隔离特性的虚拟化技术 （如硬件辅助虚拟化） 或容器化技术（如使用Kata Containers等安全容器运行时） ；为每个租户分配独立的、受配额限制的计算资源池、 存储卷和⽹络命名空间；设计并实施细粒度的资源访问控制策略， 防⽌跨租户的未授权资源访问。 n 安全功能: 持续监控跨租户的资源访问行为和潜在的隔离边界突破尝试， 利用异常检测技术及时发现可疑活动并触发告警；对关键的共享资源（如Hypervisor、容器 编排管 理节点）进行严格的安全加固和访问控制。

  • n 技术验证与渗透测试：组织专业的安全团队进行有针对性的跨租户渗透测试， 尝试从一个租户的环境中访问或影响另一个租户的资源、 数据或应用， 全面检验隔离机制的有效性和健壮性； 仔细审查虚拟化平台、 容器编排系统以及⽹络设备的隔离配置和访问控制策略的正确性和完整性。 n 配置审计与合规检查: 定期对多租户平台的资源隔离配置进行全面的审计， 确保其符合既定的安全策略、行业最佳实践和相关的合规性要求（如SOC 2对多租户环境的隔离要求） 。

5.6 供应链安全风险（与运行环境强相关部分）

AI运行环境的安全性深受其复杂的供应链环节影响， 从硬件采购、 软件预装、 更新分发到依赖的第三方服务乃至外包人员， 都可能引入潜在风险。 此类风险包括设备出⼚时预装的软件/固件中可能存在的恶意后门、不安全的更新与分发机制被利用以植入恶意代码、核心硬件组件在漫长的供应链中被篡改或植入硬件⽊马，以及依赖的第三方外包人员或服务提供商因自身安全管理不善而引入的风险。这些来自供应链的威胁通常具有⾼度的隐蔽性和破坏性，对AI系统的底层信任构成严重挑战。

5.6.1 预装软件/固件后门
概要说明

5.7 日志与监控安全风险

全面、准确、且安全可靠的日志记录与实时、有效的安全监控是AI系统安全运营和事件响应的基石。此类风险主要包括AI系统及其运行环境的日志记录不充分、格式不统一或易于被篡改， 导致安全事件发生时难以进行有效的追溯、 取证和根因分析；用于安全监控的系统（如SIEM、APM、IDS/IPS）自身存在安全漏洞被攻击者利用，从而规避检测、隐藏攻击痕迹或破坏监控能力；以及由于告警规则配置不当、阈值设置不合理或AI系统行为的复杂性， 导致产生大量的误报警或低价值告警， 淹没真实的安全威胁， 引发安全运营团队的“告警疲劳”。

5.7.1 日志记录不充分或易被篡改
概要说明
AI系统及其运行环境（包括操作系统、⽹络设备、应用程序、AI模型推理引擎等）产生的系统日志、 应用日志、 安全审计日志等记录不充分 （例如， 关键事件未记录、 记录字段不全、时间戳不准确） ， 或者日志存储和管理机制缺乏有效的完整性保护措施 （如防篡改、 防删除） ，导致在发生安全事件或系统故障时，难以进行有效的事件追溯、原因分析和数字取证。

  • n 管理制度：制定并严格执行全面的日志记录策略和标准， 明确规定需要记录的关键事件类型、 核心字段、 保存周期以及日志格式；确保所有日志都具备准确的时间戳， 并与可信的时间源同步；强制要求日志具备⾼度的完整性和不可否认性。 n 架构设计: 优先采用集中的、 经过安全加固的日志管理和分析系统 （例如，SIEM平台） ；对存储的敏感日志内容（如包含PII或操作细节的日志）进行强加密保护。 n 安全功能: 对所有重要的日志文件采用数字签名技术或将其存储在写一次读多次（WORM）的 介质 上 ，以 防 ⽌未经授权的 篡改 或 删除；实施 严格的 、基于 角 色 的 日志访问控制策略，确保只有授权人员才能查阅和管理日志。

5.7.3 告警疲劳与误报
概要说明
AI安全监控系统由于告警规则配置过于宽泛或不精确、告警阈值设置不合理、缺乏有效的告警关联分析和智能优先级排序机制，或者由于AI系统本身的复杂性和行为模式的⾮确定性，可能在短时间内产生大量不准确的（即误报，False Positives）或价值较低的告警信息。这会导致安全运营团队（如SOC分析师）不堪重负，产生“告警疲劳”（Alert Fatigue）， 难 以从海量的告警事件中及时、 准确地发现和响应真实的安全威胁和紧急事件， 从而降低了整体安全监控体系的实际有效性。

6.1.2 AI伦理与偏见放大
概要说明
AI系统由于其训练数据中潜藏的偏见、算法设计本身可能存在的歧视性逻辑，或者在与用户交互过程中学习到并放大了不当的价值观， 导致其在决策、 推荐或内容生成时， 对特定人群（基于种族、性别、年龄、宗教、性取向、社会经济地位等敏感属性）产生不公平、歧视性或侮辱性的对待。 这不仅可能引发严重公平问题和用户抵制， 还可能导致企业面临法律诉讼和声誉危机。

6.1.3 模型可解释性不⾜导致的风险追溯困难
概要说明
许多先进的AI模型（特别是深度学习模型）由于其内部结构的复杂性（即“⿊箱”特性） ，其做出特定决策或预测的内部逻辑和具体原因往往难以被人类完全理解和解释。这种可解释性的缺乏， 使得在AI系统产生错误、 不公平或有害的输出时， 难以准确地追溯问题的根源、定位缺陷所在、界定相关责任，也给模型的调试、改进、合规性证明和获取用户信任带来了 巨大挑战。

C 法务合规团队 （用于满⾜监管机构对决策过程透明度的要求） 、 业务领域专家（帮助理解模型解释与业务逻辑的关联） 、安全事件响应团队（在发生AI相关事件时进行原因调查）

控制措施
  • n 管理制度：根据AI应用的不同风险等级和具体业务场景，制定并执行差异化的模型可解释性要求和评估标准；在AI模型的整个开发和迭代过程中，强制要求记录关键的设计选择、训练过程、超参数配置以及影响模型行为的重要因素。 n 架构设计: 在模型选型和设计阶段， 应在满⾜性能要求的前提下， 优先选择或设计具有较⾼固有可解释性的模型（例如，线性模型、决策树、规则列表等） ，或者积极研究并采用先进的、与模型无关的（model-agnostic）事后可解释性方法（如LIME、SHAP、Grad-CAM等） ；建立完善的模型文档和元数据管理系统，详细记录模型的输入特征、输出含义、训练数据描述、性能指标以及已知的局限性。 n 安全功能: 集成或开发专业的、用户友好的可解释性AI工具和可视化界面，能够为不同类型的用户（如开发者、审计员、业务用户、最终用户）生成针对模型具体决策的、易于理解的解释报告或可视化展示；对于所有关键的AI系统决策过程，要求进行详细的、 可供事后审计和追溯的日志记录， 包括输入数据、 模型版本、 关键中间参数和最终输出。
验证措施
  • n 技术验证与评估：使用标准化的可解释性评估指标（如解释的保真度、一致性、简洁性等）和专业的评估工具，对AI模型所采用的可解释性方法的实际效果和可靠性进行系 统性评估； 通过模拟发生安全事件或模型产生错误决策的场景， 严格检验现有日志记录和可解释性工具对于事件原因追溯和责任认定的⽀持程度。 n 文档与流程审查: 仔细审查模型文档的完整性、 准确性和清晰度， 确保其能够充分解释模型的工作原理和行为特征。 n 用户反馈与可用性测试: 收集不同类型的用户（特别是业务用户和受AI决策影响的个体）对模型解释的理解程度、接受度和实际满意度的反馈意见。

6.5.4 智能体及MCP服务情报监控和应急响应
概要说明
在开放的AI智能体及MCP服务生态市场中， 仅仅进行准入时的安全评估和基线检查是不⾜够的。由于新的漏洞可能被发现、攻击手法不断演进、服务提供商的配置可能发生变更，或者市场中出现利用合法服务进行恶意活动的趋势，因此必须建立针对市场内应用和服务的持续性威胁情报监控、 漏洞信息共享以及快速有效的应急响应机制。 缺乏这些机制， 将使得整个生态系统在面对新兴威胁时反应迟缓，难以有效保护用户和参与者的利益。
